### Wave 5 UX bug diagnosis (no fix shipped yet)

Two root causes identified:

1. **Invisible placeholders.** `ThumbnailCellView` line 36 uses `Theme.secondaryBackground` (0.15 grey) for unrendered cells. Strip background (`ThumbnailStripView` line 72) is *also* `Theme.secondaryBackground`. Identical color → unrendered cells are literally invisible. The strip looks empty everywhere except the few cells the renderer has finished — exactly what the screenshot shows.

2. **Render task churn during scroll.** `StripModel.scheduleRender()` (lines 140–168) cancels the prior task and waits 80 ms before kicking off. During an active scrub, `onScrollGeometryChange` fires every frame → `onVisibleRangeChange` → `scheduleRender` → cancel + new 80 ms debounce. The task never starts until the user has been still for 80 ms straight. So during continuous scrubbing, no rendering happens at all.

Fix proposal (pending user OK):
- **(a)** Placeholder visually distinct from strip background — darker shade (`Color(white: 0.07)`) + 1 pt hairline border so the empty grid is *visible* and reads as "loading". ~5 lines in cell view.
- **(b)** When `onScrollPhaseChange` flips `isScrolling` false, bypass the 80 ms debounce — kick off render immediately. Keep the debounce only for the "still scrolling" case. ~10 lines in `StripModel.scheduleRender`.

## Decisions

- **Modifier-key detection: `onModifierKeysChanged` + synchronous `NSEvent.modifierFlags` fallback, both routing to `onCmdTap`.** No double-fire risk because the tap gesture fires once; the in-closure NSEvent read covers focus-race cases where the change handler hasn't yet reflected the current modifier set.
- **Half-cell-interval pin window computed in seconds, not CMTime.** `abs(CMTimeGetSeconds($0) - cellTimeSec) < 0.5 / model.density`. CMTime arithmetic via `CMTimeSubtract` + `CMTimeAbsoluteValue` would work but Double seconds is cleaner since `activeIndex`, `thumbnailTimes` etc. already work in seconds. Pin resolved at the strip level — cell stays vm-unaware.
- **Bare-`M` scoped to strip focus, not app-wide.** App-wide via `.keyboardShortcut("m", modifiers: [])` would swallow `M` typed into the right-pane interval TextFields. The strip is where the user has focus while scrubbing anyway, so the trade-off is in our favor.
- **Manual list sorted on insert (Option A) over keeping capture order on disk (Option B).** Right-pane list IS the destination preview — row N = filename `_N`. Option B (drop the sort in `TimeListGenerator`) would have preserved capture-history in the UI but lost the `images(for:)` sequential-GOP-decode perf benefit upstream and would be less natural for "give me PNGs of the interesting moments in storyline order."
- **Wave 5 considered quantitatively passing.** Even though UX bugs surfaced, the strict memory ceiling check (566 → 700 MB) is satisfied with 33% headroom and the saw-tooth eviction pattern confirms NSCache is doing its job. The UX issues are *render-pipeline scheduling* and *placeholder visibility* bugs, not memory bugs.

**Player-seek → strip auto-scroll (added on user request, ~25 lines)**

User reported "clicking in the player on a different time does not make the thumbnails jump to that time." Implemented via:

1. Added `@State private var seekScrollTarget: Int? = nil` to `ThumbnailStripView`.
2. Wrapped the `ScrollView` in `ScrollViewReader { proxy in … }` so we can call `proxy.scrollTo(_:anchor:)` programmatically.

### Decisions

- **Placeholder color: `Color(white: 0.07)`, no border.** A darker shade alone is sufficient when paired with the existing 1 pt cell-spacing gap (which shows the strip's 0.15 background through as dividers). Adding a `strokeBorder` overlay was considered but unnecessary — the contrast between 0.07 cells and 0.15 gaps already gives the grid a visible structure, and overlay borders would re-introduce hit-testing concerns (would need `.allowsHitTesting(false)`).
- **Debounce only while actively interacting; otherwise render immediately.** Gated on `isScrolling || isMagnifying`. The flag-state-at-task-spawn is the simplest version that handles all four cases correctly: (i) during scroll — flag true → debounce → cancelled by next tick (no waste); (ii) scroll ends — flag flipped false BEFORE the post-scroll `scheduleRender` call, so the new task sees false → skips sleep; (iii) during pinch — same as scroll; (iv) pinch ends — same. No need for `scheduleRender(immediate:)` parameter or two-task split.
- **Seek detection threshold: 0.5 s on the 1/30 s periodic time observer.** Natural per-tick delta during playback is bounded by the observer interval (≤33 ms at 1x speed; faster only at >15x rates the AVPlayer transport doesn't expose to us). 0.5 s is well above any legitimate playback delta while still catching even short seeks (e.g. tapping ⌥-arrow to step ±5 s). Lower thresholds risk false-positives during stutter; higher thresholds miss small jumps.
- **`ScrollViewReader` + `.id(idx)` over `.scrollPosition(id:)`.** `.scrollPosition(id:)` (macOS 14+) writes back the current position to its binding as the user scrolls, which would mean every scroll tick fires a state update on the strip. The reader-based pattern is one-way — we tell SwiftUI "scroll there now" exactly when needed, no chatter. The cost is having to wrap the ScrollView in another layer; brace-counting bug above is the only real friction.
- **Auto-scroll on every seek including the strip's own click-to-seek (don't try to suppress the self-recenter case).** Considered tracking "did *we* initiate this seek?" via a flag and suppressing the scroll when so, but: (i) it adds state and timing fragility, (ii) the cosmetic recenter after click-to-seek is actually pleasant — keeps the clicked thumb in view rather than at the edge of viewport. Simpler and arguably better.
- **Long-clip optimization explicitly out of scope.** Recorded in PROJECT_STATE under "What this is." The mitigation menu (hot+cold task split, coarser density at high duration, virtualized rows) is documented in Key Decisions but explicitly not on the roadmap. If a future user requests long-clip support, this is where to start.

## Decisions (executed — should be promoted to `docs/decisions.md`)

The planning session above already decided each of these; this section just records that they survived contact with code.

- **WebP omitted** — runtime probe + CropBatch dep audit verified during planning; nothing in the implementation changed that decision. Load-bearing for future-me: if someone asks "why no WebP?", the answer is in `POLISH_PLAN_post_phase5.md` Item 2 + `docs/decisions.md` (pending promote).
- **`.jpg` not `.jpeg`** in `ExportFormat.fileExtension`. Confirmed correct in practice — exported files came out as `*.jpg`.
- **Single shared quality** across JPG and HEIC. No per-format split. HEIC files at 0.85 came out reasonable in smoke; no need to revisit yet.
- **PNG default**, preserves prior behavior on first launch — confirmed via fresh-install path (would only happen on a clean Preferences); manual: PNG was selected by default in the UI even after the format change landed.
- **Quality slider hidden, not disabled**, when PNG selected. Implemented as `if vm.exportFormat.supportsCompression { ... }`. Smoke: slider vanishes/reappears as the user toggles formats. Layout doesn't jump much because the row is small.
- **Format section between Output and Options** (vs. inside the existing options DisclosureGroup). Picked this slot during implementation — Mode → Params → Output → Format → Options reads as "what / when / where / how" in that order. The Format choice doesn't belong nested inside "Options" because it materially affects file size, not just appearance.
- **xcodegen regen IS required for new files in this repo.** Promoted to project knowledge — `POLISH_PLAN_post_phase5.md`'s "auto-discovers" claim was wrong, observed empirically. The sources glob is resolved at generate time. Add to `docs/decisions.md` under "Project Foundation" so the next polish session doesn't hit the same build failure.
