  - Explore agent → archaeology on `_Published/CropBatch` + `_Published/VideoWallpaper` for proven patterns (batch Task, progress, collisions, autoreleasepool)
  - Web agent → 2025-era community wisdom on async batch + strict-concurrency
- Synthesized findings into `IMPLEMENTATION_PLAN.md` at project root — 7 tasks across 4 waves.

### Decisions (locked into the plan)

1. **Batch API: use `AVAssetImageGenerator.images(for: [CMTime])`** — the AsyncSequence introduced in macOS 13. The older `generateCGImagesAsynchronously` callback API **crashes** under Swift 6 strict concurrency (Apple DTS forum thread #761194). The serial `image(at:)` loop is safe but throws away decoder-reuse perf.
2. **Concurrency shape: `@MainActor @Observable` view-model owning one `Task`.** No custom actor (Hudson's guidance), no AsyncStream — progress flows via `await`-ing a `@MainActor` method on the model per iteration.
3. **Cancellation: `withTaskCancellationHandler { cancelAllCGImageGeneration() }` + `Task.checkCancellation()`.** Cancellation surfaces partial results — frames already on disk stay there.
4. **TaskGroup parallelism rejected** for frame extraction (unlike CropBatch's image crops). `images(for:)` exploits sequential GOP decoding internally; parallel generators would hurt throughput.
5. **Times sorted ascending** before passing to `images(for:)` — out-of-order kills decoder reuse. Filename index preserves *requested* order.
6. **Filename collisions: lift CropBatch's `appendNumericSuffix`** (`_Published/CropBatch/.../ExportSettings.swift:318–334`) verbatim. Never overwrite.
7. **`autoreleasepool` only around the PNG encode** (`NSBitmapImageRep`/`CGImageDestination` are Obj-C). Skip elsewhere — Swift Concurrency wraps jobs in pools already.
8. **Default tolerance `.zero`** (exact frame) but configurable on `ExtractionRequest` so it can be relaxed later without API churn.
9. **Scope split: pipeline now, production UI later.** Wave 3 is a minimal smoke-test `ContentView` shim — the segmented picker / polish UI is a separate plan.

## Future ideas (deferred, not in current plan)

### Thumbnail strip with zoom-density coupling
**Idea (recorded 2026-05-27):** below the AVPlayer on the left pane, show a horizontal thumbnail strip of the video. Default density is light — e.g. **1 thumbnail per second** — so opening a long clip isn't taxing on decode/memory. User can **zoom** the strip; as they zoom in, two things happen *together*:
1. **Density increases** — more thumbs per second (eventually approaching per-frame at max zoom).
2. **Thumbnail size grows** in small increments — the assumption is that zooming = searching for a specific frame, and bigger thumbs let the user spot the target faster.

**Rendering strategy:** generate thumbs lazily. When the user is *zooming in* (not scrolling), the app can opportunistically render not-yet-visible thumbs in the background — that's when extra detail is most valuable. When the user is *scrolling*, they're in transit; prioritize visible-window thumbs only and let off-screen ones come in as they enter view.

**Why it's not in this phase:** Phase 4's acceptance criteria explicitly say "thumbnail strip with zoom-controlled stride" is deferred, but the architecture must not preclude it. The current `ExtractionMode` keeps selection as `[CMTime]` and `ExtractionPipeline` already uses `AVAssetImageGenerator.images(for:)` — exactly the right shape for a future strip that consumes the same batch API.

**Hooks to consider later:**
- Reuse `AVAssetImageGenerator.images(for:)` with a `[CMTime]` derived from the visible-window × zoom-density product
- A small per-clip thumbnail cache keyed by `(time, targetSize)` so re-zooming doesn't re-decode
- "Capture this frame" on a strip thumb tap (mirrors the player's ⌘C path) — would let users build the manual list from the strip too

### Decisions (logged here; key ones to be promoted to docs/decisions.md when phase 5 begins)
- **`MagnifyGesture` attached to the parent VStack, not to ScrollView children** — empirical workaround for the macOS 15 regression of `.highPriorityGesture` and gesture-vs-scroll conflicts (Apple Forums #760035, #794212; HWS forum #28798). Single state flag `isMagnifying` flips on/off via `.onChanged`/`.onEnded`.
- **Two AVAssetImageGenerator instances** — one for the strip (relaxed tolerance ±0.5s when sparse, `.zero` when dense; `maximumSize` = 480 pt × 2x), one for the export pipeline (untouched). Reason: tolerance + maximumSize live on the generator, and `cancelAllCGImageGeneration` is global per instance — a strip cancel must not nuke an export.
- **Cache key bucketed by width** — `struct ThumbKey { timeMillis: Int64; widthBucket: Int }`, buckets `{60, 120, 240}` pt. Display width is continuous (60-240); cache fetches the smallest bucket ≥ display size, downscaled at draw time. So zoom-wiggle *within* a bucket band is free.
- **`CMTime` normalized to `timeMillis: Int64`** — `CMTime` is `Hashable` on macOS 13+ but exact-equality semantics (epoch + timescale) make it a footgun as a dict key. Normalize once at insertion. Also dodges the pre-macOS-13 crash documented in swiftlang/swift#65063.
- **Prefetch policy matrix** keyed by `(isMagnifying, isScrolling)`:
  - `isMagnifying`=true → visible ± 50% width (speculative)
  - `isScrolling`=true → visible only
  - both false (idle) → visible ± 20% width
- **Zoom curves**: width `60 + 180·smoothstep(0,10,zoom)` (saturates at zoom 10); density `pow(2, zoom·0.6)` capped at `nominalFPS`. So effective zoom range is ~10 levels — saturates earlier than expected, but bucketing math still applies because density continues changing.
- **Click = seek; ⌘-click = capture** to `vm.manualTimes`. New `vm.captureFrame(at:)` mirrors existing `captureCurrentFrame()` (line 114) but for arbitrary times. Pins rendered as 4 pt accent dots over their cells.
- **NSCache tiered by `ProcessInfo.physicalMemory`**: 200 MB on ≤8 GB Macs, 300 MB on 16 GB, 500 MB on ≥32 GB. Per-entry cost = `cgImage.bytesPerRow * cgImage.height`.
- **80 ms debounce + cancel-replace** on zoom-driven render scheduling — prevents pile-up when the user pinches rapidly.
- **Storage format**: `CGImage` only for now. JPEG-Data secondary tier deferred unless profiling shows pressure.

### Reality-check on bucket design (raised by user)
User asked: "100 zoom steps but only 5-10 rendered sizes, so 10-20 zoom steps share a cached size?" Answer: closer to 3 buckets across ~10 effective zoom levels = each bucket covers ~3-4 levels. *Important nuance:* bucketing saves re-decodes on SIZE only — DENSITY changes continuously, so new `CMTime`s are fetched at every zoom step regardless. Bucketing's big wins are (a) zoom-wiggle within a band, (b) return-to-previous-zoom. Could bump to 5 buckets if intermediate-zoom sharpness looks bad in practice.

### Findings worth flagging (non-blockers)
- `TapGesture.modifiers(.command)` has reported flakiness on macOS 15 — Wave 4.2 has a fallback path noted (NSEvent.modifierFlags via tiny NSViewRepresentable overlay).
- **No shipping app** couples zoom→size + zoom→density on a single gesture: iMovie has density-only, FCP has size-only. Design is genuinely novel relative to references.
- `images(for:)` Swift-6-safe path is *the* one to use (DTS #761194 crash is in the legacy callback API only). Already proven in `ExtractionPipeline`.
- macOS 15's `ScrollPhase` + `isScrolling` maps **exactly** onto "zoom = looking, scroll = transit" — no need for custom NSEvent intercepts.

## Phase 4.5 polish — unload-gap fix + `/minimums` audit

### Unload-video gap (DONE)

**Root cause analysis:** `vm.load(_:)` already atomically replaces the loaded video (clears `manualTimes`, resets `player`/`metadata`/`sourceURL`). `ContentView.swift`'s root `.videoDropTarget` already calls `vm.load(...)` unconditionally on any new video drop. **The drop modifier never sees drops over the player area** — AppKit's `AVPlayerView` is an `NSView` with its own drag/drop hit-testing that intercepts file URLs before they reach SwiftUI's `onDrop`. So drop-to-replace works in theory, but only over the non-player parts of the window — invisible and undiscoverable.

**Fix shipped:**
- `App/ExtractionViewModel.swift` — added `clear()` method. Cancels any in-flight `job`, pauses the player, nils out `sourceURL`/`player`/`metadata`, clears `manualTimes`/`progress`/`statusMessage`/`lastError`, sets `isRunning = false`. Output folder + overlay/numbering options preserved intentionally (a user-friendly choice — they don't want their settings nuked just because they swap clips).
- `UI/LeftPaneView.swift` — added a small `xmark.circle.fill` button at the leading edge of the loaded-state footer, plain button style, secondary tint, `.help("Remove video")`.

### Decisions

- **Folder layout deviation from plan.** Plan file scattered the 5 Phase 5 files across `Core/` (ThumbnailRenderer), `App/` (ThumbnailCache, StripModel), and `UI/` (cell, strip). PROJECT_STATE "Next Actions" already directed them into a new `Thumbnail/` folder, and an empty `Thumbnail/` directory was sitting in the source tree as a signal. Chose single-feature folder — five files all serving one feature reads more naturally grouped than scattered by architectural category. Plan file edited to record the deviation. No code-side cost; xcodegen handles either layout.

- **Per-`render(...)` generator instance, not one shared at init.** ThumbnailRenderer builds a fresh `AVAssetImageGenerator` inside each `render(...)` call. `AVAssetImageGenerator` is cheap to construct; isolating per-call state lets us vary `maximumSize` and `tolerance` per zoom level without mutating shared state, and means a strip-cancel can't accidentally interfere with the export pipeline's generator (which is fully separate anyway).

- **No manual `cancelAllCGImageGeneration` in renderer.** `AsyncStream.onTermination` cancels the producer Task when the consumer drops its iterator; cancellation propagates into `images(for:)` automatically (same path the export pipeline relies on). The shim is only needed because `AVAssetImageGenerator` is non-Sendable and we hop it into the producer Task closure.

- **`cacheVersion: Int` with `&+=` instead of plain `+=`.** Cells can't observe `NSCache` directly (it's not `@Observable`), so they re-read via `cache.image(for:)` whenever any observed state changes. `cacheVersion` is the cheapest such signal — bumped after every successful store. Overflow-trapping `+=` would crash on long sessions at high zoom (thousands of thumbs per minute); `&+=` wraps silently — harmless because consumers care about "did it change" not "what value is it".

- **Debounce + cancel-replace inside `scheduleRender`, not at view layer.** Each model-mutation call (`onZoomChange`, `onScrollPhaseChange`, etc.) calls `scheduleRender()` immediately, which cancels the prior task and starts a new one with an 80 ms `Task.sleep` head. Rapid event bursts (pinch deltas, scroll geometry updates) collapse into a single render kicked off only after the user pauses. Avoids piling up generator work on the AVF side. Captures all derived values (`bucket`, `tol`, `range`, `allTimes`) on the main actor before the Task hop so the closure stays `Sendable`-compliant under Swift 6.

Hoisting to App level would also require hoisting that lifecycle. Cleaner: `LeftPaneView` publishes a `StripZoomActions` struct via `.focusedSceneValue(\.stripZoom, …)`; `ZoomCommands` reads via `@FocusedValue` and auto-disables when nil.
- **`.onKeyPress` overload disambiguation.** Single-key form `.onKeyPress(_:)` defaults to a zero-arg action closure — to receive a `KeyPress` argument (needed for modifier filtering), pass `phases: .down` explicitly OR use `.onKeyPress(keys: [...])`. Latter only has the `(KeyPress) -> Result` overload, so it disambiguates cleanly. Chose `keys:` form throughout for consistency.
- **⌘+scroll-wheel via NSEvent local monitor, not NSViewRepresentable.** Considered NSViewRepresentable overlay with `nextResponder?.scrollWheel(with:)` pass-through but it's fragile in SwiftUI's responder chain. `NSEvent.addLocalMonitorForEvents` returns the event by default (transparent pass-through), returns `nil` to consume — scoped to the strip's `.task` lifetime, monitor removed in `defer`.
- **Cell-side fallback over centralized cache changes.** The cascade lives in `ThumbnailCellView` because cells already know their `(time, bucket)` identity and can hold a `@State` last-image without touching the cache contract. Alternatives considered: snap timestamps to a fixed 100ms base grid (only helps one zoom direction); centralized nearest-time lookup in the cache (NSCache doesn't support iteration, would need a parallel index).

### Operational learnings

- **SourceKit "Cannot find type" errors are 100% noise here.** Same pattern as Wave 2 — every edit to the strip files triggered a wave of false positives referencing in-flight Swift files. Trust `xcodebuild`.
- **`MagnifyGesture.value.magnification` is absolute, not delta.** Worth remembering for any future continuous gesture work (Rotate, Drag have similar absolute-vs-delta gotchas). Pattern: snapshot starting state in `.onChanged`'s first tick, apply gesture value against the snapshot.

**User reported the click "kinda works but selects the thumb on the left of it, off by 50px or something."**

Root cause traced in ~30 s of arithmetic, not geometry. At default zoom: `density = pow(2, 0.6) ≈ 1.516`. Cell 1's time is `1/1.516 ≈ 0.66 s`. AVPlayer's sample-accurate seek with `.zero/.zero` tolerance lands on the frame *containing* 0.66 s — at 30 fps that's frame 19, whose start time is `19/30 ≈ 0.633 s`. `currentTime()` reports the frame's *start*, not the seek target. Then `activeIndex` used `.rounded(.down)`:

```
floor(0.633 × 1.516) = floor(0.96) = 0   ← cell 0, not cell 1
```

Visual offset: "one cell to the left," ≈ cellW pixels — matches the user's "50px or something" report. The click was always going to the correct cell; the border indicator was lying about it.

Fix: `.rounded(.down)` → `.rounded()` in `ThumbnailStripView.activeIndex`. With round-to-nearest, the ε undershoot from sample-accurate seek now rounds back up to the clicked cell. Rebuilt, user confirmed: "works now."

### Decisions

- **`.rounded()` for `activeIndex`, not `.rounded(.down)`.** Semantically `.rounded(.down)` was "the cell whose time range contains currentTime" — correct in continuous-math terms. But AVPlayer's sample-accurate seek snaps to the nearest decoded frame's start time, which generally undershoots the seek target. Round-to-nearest gives "the cell whose stored timestamp is closest to currentTime" — robust to ε undershoot, plus during natural playback the border advances at each cell's midpoint instead of the end (subjectively feels more responsive). Trade-off: at the boundary between two cells the border picks the higher-indexed one, which is a slight semantic shift, but invisible in practice.
- **Keep `.zero/.zero` seek tolerance** despite the activeIndex headache. Considered switching to `model.tolerance` (which is `±0.5s` at low zoom) to match the strip's own renderer tolerance, but sample-accurate seek is the right contract for click-to-seek — the user clicked a specific thumb, they should land on its frame, not the nearest keyframe minus 0.5 s. The activeIndex fix solves the visual mismatch without compromising seek precision.
- **`.allowsHitTesting(false)` on the active-cell border overlay** — pre-emptive fix based on the SwiftUI hit-testing research. Even though `strokeBorder` paints only a thin line, its hit region is the full layout rect. Without the modifier, taps on the active cell would have been swallowed by its own border overlay. Decorative overlays = `.allowsHitTesting(false)`, as a rule.
- **`onTap` is required (no default), not optional.** Cell prop is `let onTap: (CMTime) -> Void`, not `var onTap: (CMTime) -> Void = { _ in }`. Surfaces missing wiring at compile time. Trivial price (the single call site has to pass something) for catching bugs at build rather than runtime.

### Operational learnings

- **Trust the floating-point arithmetic, not the layout debugger.** User said "off by 50px." Geometry/hit-region was the obvious suspect, especially after we just added `.allowsHitTesting(false)` and `.contentShape(Rectangle())` (modifiers that exist *because* hit regions can go wrong). But the cell + grid math was solid — the bug was in a single `.rounded(.down)` two hundred lines away. Lesson: when the visual offset is *exactly* one unit-of-something (cellW here), check unit-quantization first before geometry.
- **Sample-accurate AVPlayer seek lands at-or-before the target.** Not exactly the target, not the nearest frame — the frame *containing* the target time, reported by `currentTime()` as that frame's start. Worth remembering for any future strip/timeline UI: if you derive UI state from `player.currentTime()` after a seek, expect ε undershoot when the seek target isn't frame-aligned.
- **Plan-first when MCPs and web are cheap.** Spent ~2 min on parallel `sosumi` + `apple-doc-mcp` + WebSearch before touching code. Caught the `.allowsHitTesting(false)` fix that would have been a second user-reported bug otherwise. Also caught the `TapGesture.modifiers(.command)` flakiness in time to redirect Wave 4.2's spec before writing any code for it. Plan drift is the cheapest bug to fix.
