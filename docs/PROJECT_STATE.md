# PROJECT_STATE — ScreenshotFromVideos

## What this is
macOS app that extracts still frames from a video as PNGs — every Nth frame or user-selected timestamps — with optional timestamp overlay and chronological filename numbering. Built primarily to feed Claude Code visual context from screen recordings.

**Design target: short clips, 30 s to 2 min.** Longer videos (10+ min) work but the thumbnail-strip rendering is visibly slower since the asset gets walked through more keyframes per scroll. Optimizing for multi-hour clips is explicitly out of scope.

**User-facing name:** "Stills From Video" (since 2026-05-27 — see Key Decisions). **Internal identifiers** (bundle ID `com.lucesumbrarum.ScreenshotFromVideos`, repo folder, Xcode target, scheme) intentionally kept as `ScreenshotFromVideos`.

## Key Decisions
- Build tooling: **xcodegen** (`01_Project/project.yml`) — regenerable and reviewable
- macOS 15.0+ / Swift 6.0 / SwiftUI / notarized non-sandboxed
- Bundle ID: `com.lucesumbrarum.ScreenshotFromVideos`, team `FDMSRXXN73`
- **Lift strategy:** distill, don't copy whole. Each lifted file ≤120 lines, single-purpose, with a header comment naming its source
- **Extraction mode:** segmented picker — Interval (seconds / frames) OR Manual timestamps
- **Batch frame API:** `AVAssetImageGenerator.images(for: [CMTime])` (macOS 13+ AsyncSequence). The older callback API crashes under Swift 6 strict concurrency (Apple DTS #761194)
- **Concurrency shape:** `@MainActor @Observable` view-model owning one `Task<Void, Never>`; per-iter `@MainActor` progress callback; no custom actor, no AsyncStream
- **Cancellation:** `withTaskCancellationHandler { cancelAllCGImageGeneration() }` + `Task.checkCancellation()`. Partial PNGs preserved on cancel
- **TaskGroup parallelism rejected** for frame extraction — `images(for:)` exploits sequential GOP decoding internally
- **Times sorted ascending** before passing to `images(for:)` (in `TimeListGenerator.times(for: .timestamps(_:))`); filename indices are then assigned in delivery order, so on-disk numbering is chronological by clip time. The right-pane manual list also sorts on insert (since 2026-05-28) so the list row order matches what export writes
- **Filename collisions:** `_1`, `_2`, … suffix pattern; never overwrite
- **`autoreleasepool` only around the PNG encode step**; Swift Concurrency handles the rest on Darwin
- **Default tolerance `.zero`** but configurable per request
- **(Wave 2)** Discrete UI fields (`tab/intervalUnit/intervalSeconds/intervalFrames/manualTimes`) are the source of truth; `buildRequest()` resolves them into the locked `ExtractionMode` at Export time. No redundant `mode` field on the view-model
- **(Wave 3)** Live current-time label uses `TimelineView(.periodic)` over `addPeriodicTimeObserver` — no observer token to manage, no leak risk
- **(Wave 4)** Skipped `.environment(\.theme, .dark)` from the plan — `Theme` is a static enum, not an `EnvironmentKey`. Views read `Theme.primaryBackground` directly. Revisit if/when user-customizable accents are needed
- **(Wave 4)** `SplitViewAutosaveHelper` lifted verbatim from cookbook `01-window-layouts.md`. Divider position persists via `NSSplitView.autosaveName = "MainSplit"`
- **(Phase 4.5)** **App renamed user-facing to "Stills From Video"**. Internal identifiers (bundle ID `com.lucesumbrarum.ScreenshotFromVideos`, repo folder, Xcode target, scheme) kept. Only `CFBundleName` and `CFBundleDisplayName` changed. Rationale: "Screenshot" implies screen capture but the app accepts arbitrary video files; "Still" is the established cinematography term for an extracted frame
- **(Phase 4.5)** **Preferences pattern:** dedicated `App/Preferences.swift` enum with `UserDefaults` read/write static funcs. View-model properties declared without default values, populated in `init()` from `Preferences.*()`, write-throughs via `didSet`. The no-default trick avoids spurious writes during init (Swift skips `didSet` on first assignment in init). `nonisolated(unsafe) static let defaults = UserDefaults.standard` is the Swift-6 strict-concurrency escape hatch — `UserDefaults` is documented thread-safe but Foundation doesn't conform it to `Sendable`
- **(Phase 4.5)** **`@State var vm` lives at App level**, not inside ContentView. Required for the `Commands { ... }` block on `WindowGroup` to capture the same view-model instance. Single-window app — multi-window would need `@FocusedValue` instead
- **(Phase 4.5)** **Menu bar uses `CFBundleName`, not `CFBundleDisplayName`.** To rename the user-facing app cleanly, set both bundle strings to the same value. `PRODUCT_NAME` / bundle ID / target name don't need to match — those are internal
- **(Phase 4.5)** **No ⌘⇧O for output folder.** The right-pane Choose folder button is right there and users set the output folder once per session — a menu duplicate would be redundant
- **(Phase 4.5)** **Icon strategy: placeholder generator + Claude Design prompt** in `02_Design/`. The placeholder is reproducible (`swift 02_Design/generate_app_icon.swift`); the polished version is one prompt away. Placeholder visual: dark navy gradient + filmstrip (sprocket holes punched via even-odd fill) + a detached warm-amber frame above-right (the extraction gesture)
- **(Phase 5 Wave 2)** **Contact-sheet grid over horizontal filmstrip.** Original plan called for a single-row filmstrip (iMovie/FCP precedent). User asked for multi-row wrapping after seeing the empty vertical space; `LazyVGrid(.adaptive)` with active-cell border replaces `LazyHStack` + vertical playhead. The grid pattern matches QuickTime's keyframe-export UI and the Photos browser — also valid prior art
- **(Phase 5 Wave 2)** **Player ↔ strip split is user-adjustable.** Wrapped both in a `VSplitView` with `autosaveSplitView(named: "LeftPaneVSplit")`. Player `minHeight: 120`, strip `minHeight: 100, idealHeight: 320`. Footer stays outside the split, below both. Same `SplitViewAutosaveHelper` lifted from cookbook 01-window-layouts.md works for vertical splits too
- **(Phase 5 Wave 2)** **`.frame(maxWidth: .infinity)` on vertical scroll containers inside `VSplitView`.** `ScrollView(.vertical)` containing a `LazyVGrid` reports a narrow intrinsic width (= one cell), and `VSplitView` matches both panes to it. Without an explicit horizontal-fill modifier on the scroll container, the entire split column collapses to that narrow width. Worth remembering for any future split-view + vertical-scroll combos
- **(Phase 5 Wave 3)** **MagnifyGesture: snapshot the baseline, don't compound.** `MagnifyGesture.value.magnification` is the **absolute** scale from gesture start (1.0 at start, grows or shrinks). Apply it against a snapshot of `zoomLevel` taken in `onMagnifyStart`, NOT against the current `zoomLevel` (which would compound each tick and blow past the clamp). `StripModel.zoomBaseline` is the snapshot. Symptom of the wrong pattern: even tiny pinches send zoom straight to the max clamp within ~50ms. `commitZoom` also re-snaps the baseline so the next gesture starts clean
- **(Phase 5 Wave 3+)** **All zoom input paths converge on `setZoom(_:)`** — menu commands, keyboard, scroll wheel, and slider all route through this one mutator (not just `onZoomChange`, which is pinch-only). `setZoom` clamps to [1, 12] AND updates `zoomBaseline` to the new value, so a subsequent pinch starts from the user's last set point. Without that baseline sync, pinching after a slider drag would snap back to whatever was the baseline at last `onMagnifyStart`
- **(Phase 5 Wave 3+)** **`@FocusedValue` for view-scoped menu commands.** The `View > Zoom` menu items can't bind directly to `StripModel` because the model lives inside `LeftPaneView` (`@State private var stripModel: StripModel?`), not at App level. Hoisting it to App level would require also hoisting the metadata-driven lifecycle. Instead: `LeftPaneView` publishes a `StripZoomActions` struct via `.focusedSceneValue(\.stripZoom, …)`; `ZoomCommands` reads via `@FocusedValue(\.stripZoom)` and `.disabled(zoom == nil)` greys the menu out when no video is loaded. Cleaner than hoist + closure-prop
- **(Phase 5 Wave 3+)** **`.onKeyPress` overload disambiguation.** Single-key form `.onKeyPress(_:)` defaults to a zero-arg action closure; to receive a `KeyPress` argument (for modifier filtering) either pass `phases: .down` explicitly OR use `.onKeyPress(keys: [...])` — the latter only has the `(KeyPress) -> Result` overload, so it disambiguates cleanly. Chose `keys:` form throughout for consistency
- **(Phase 5 Wave 3+)** **⌘+scroll-wheel via NSEvent local monitor, not NSViewRepresentable.** Tried mentally: NSViewRepresentable overlay needs `nextResponder?.scrollWheel(with:)` pass-through, which is fragile in SwiftUI's responder chain. Cleaner pattern is `NSEvent.addLocalMonitorForEvents(matching: .scrollWheel)` in a `.task` on the strip view — returns the event by default (pass-through), returns `nil` to consume (when ⌘ held). Scoped to the strip's lifetime: the task is cancelled when the strip leaves the view tree, the monitor is removed in `defer`. Inside the closure, `Task { @MainActor in m.applyZoomDelta(dy) }` hops to MainActor for the model call
- **(Phase 5 Wave 3+)** **Cell-side fallback cascade kills the zoom-transition flash.** Cells now resolve via `bestImage()`: exact `(time, bucket)` → same time at other buckets `[240, 120, 60]` (SwiftUI's `scaledToFill` handles the resize — scaling down is visually free) → `@State lastImage: CGImage?` (the last successfully-displayed exact-match, kept across `(time, widthBucket)` rebinds via `.onChange(of: cellIdentity) { syncLastImage() }`). Alternatives rejected: snap timestamps to a 100ms base grid (only helps zoom-out, not zoom-in); centralized nearest-time lookup (NSCache doesn't support iteration → would need a parallel index). `lastImage` is cell-scoped and evaporates on LazyVGrid lazy unload, so no memory leak — ~5 MB extra resident for 100 visible cells
- **(Phase 5 Wave 4.1)** **`.rounded()`, not `.rounded(.down)`, in `activeIndex`.** AVPlayer's sample-accurate seek with `.zero/.zero` tolerance lands on the frame *containing* the target time, and reports `currentTime()` as that frame's *start* time — which is ε below the target whenever the target doesn't land exactly on a frame boundary (the common case at fractional `density`, e.g. `pow(2, 0.6) ≈ 1.516`). With `.rounded(.down)`, that ε pushed `activeIndex` one cell to the left of the clicked cell. Round-to-nearest tolerates the undershoot and matches the user's mental model ("the cell whose timestamp is closest to currentTime"). Trade-off during natural playback: the border now flips at the midpoint of each cell's time range instead of at its end — subjectively feels more responsive, not less correct
- **(Phase 5 Wave 4.1)** **`.allowsHitTesting(false)` on decorative overlays inside tap-handling views, as a rule.** SwiftUI hit-tests by **layout shape, not visual shape** (HWS + Apple DevForums). A `Rectangle().strokeBorder(...)` overlay has a full-rect hit region despite painting only a thin border line — without the modifier, taps on the active cell would have been swallowed by its own decorative border. Applies to any overlay that's purely visual
- **(Phase 5 Wave 4.1)** **Sample-accurate seek (`.zero/.zero`) is the right contract for click-to-seek, even with the activeIndex headache.** Considered switching to `model.tolerance` (which is `±0.5 s` at low zoom) to match the strip renderer's own tolerance, but the user clicked a *specific* thumb — they expect to land on its frame, not the nearest keyframe minus 0.5 s. Fix the visual-feedback math, not the seek precision. Apple's *"Controlling the transport behavior of a player"* article confirms this is the canonical pattern; the decoding-delay cost cited there only matters for continuous-scrub UIs
- **(Phase 5 Wave 4.1)** **Modifier-key detection in SwiftUI: prefer `onModifierKeysChanged(mask:initial:_:)` (macOS 15+) over `TapGesture.modifiers(_:)`.** The latter has been flaky on macOS for years (Apple DevForums #654154, ongoing 2024–2025). `onModifierKeysChanged` tracks the held state into `@State`, branched inside `.onTapGesture`. `NSEvent.modifierFlags.contains(.command)` read synchronously at tap time is the second-tier fallback
- **(Phase 5 Wave 4.2)** **Wave 4.2 used both `onModifierKeysChanged` AND a synchronous `NSEvent.modifierFlags` read inside the tap closure, belt-and-suspenders.** The state-tracked variable handles the normal case; the in-closure read covers focus-race cases where the strip ScrollView regains focus mid-tap and the change handler hasn't yet fired with the current modifier set. Both paths route to `onCmdTap` — no double-fire risk because the tap gesture fires once
- **(Phase 5 Wave 4.2)** **Manual-pin half-interval window in seconds, not CMTime.** Cell pin renders when `abs(CMTimeGetSeconds($0) - cellTimeSec) < 0.5 / model.density`. CMTime arithmetic via `CMTimeSubtract` + `CMTimeAbsoluteValue` would work but Double seconds is cleaner since we already work in seconds in `activeIndex`, `thumbnailTimes`, etc. Resolved at the strip level — cell stays vm-unaware (takes a `manualPinned: Bool`, not the model)
- **(Phase 5 — extra capture shortcut)** **Bare `M` on the focused strip routes to `vm.captureCurrentFrame()`.** Same `.onKeyPress(keys:)` pattern as the +/-/0 zoom keys, with the same `⌘/⌥/⌃` modifier-filter guards so it doesn't shadow accelerator combos elsewhere. Scoped to strip focus rather than app-wide (via Commands menu / window-level `.keyboardShortcut(modifiers: [])`) on purpose: a bare-letter app-wide shortcut would swallow `M` typed into the right-pane interval TextFields. The strip is where the user naturally has focus while scrubbing
- **(Phase 5 Wave 5)** **Placeholder must be visually distinct from the strip's own background.** Both used `Theme.secondaryBackground` (0.15 grey) → empty cells were invisible against the strip. Now placeholder is `Color(white: 0.07)`; the 1 pt cell-spacing gap shows the lighter strip background through, so the empty grid reads as "many dark cells separated by lighter dividers" → "loading", not "void". When the strip is genuinely empty (no video loaded), the strip's `Theme.secondaryBackground` fill is what shows — no placeholder cells exist
- **(Phase 5 Wave 5)** **Debounce only while actively interacting.** `StripModel.scheduleRender()` now gates its 80 ms `Task.sleep` on `isScrolling || isMagnifying`. Debounce kicks in during continuous scroll/pinch (good — prevents render thrash that was preventing *any* rendering during scrub). The moment either phase ends, the next `scheduleRender` call (triggered by `onScrollPhaseChange(false)` or `onMagnifyEnd`) reads both flags false → skips the sleep → renders immediately. End-to-end latency from scroll-release to first thumbs ≤1 frame instead of 80 ms + decode lag
- **(Phase 5 Wave 5)** **Player-seek → strip auto-scroll via delta-detection on the periodic time observer.** Natural per-tick deltas at the 1/30 s observer interval are ≤33 ms; anything >0.5 s is treated as a seek (user clicked the AVPlayer seek bar, or click-to-seek on an off-screen thumb). Setting `@State seekScrollTarget = activeIndex` then triggers a `ScrollViewReader`-backed `.onChange` that animates `proxy.scrollTo(target, anchor: .center)` over 0.15 s. Doesn't fire during natural playback (so user's manual strip scroll position is respected during playback), but does fire on every seek including the strip's own click-to-seek (gives a small "recenter after click" affordance, kept on purpose)
- **(Phase 5 Wave 5)** **Design target is 30 s–2 min clips; long-clip optimization out of scope.** Memory smoke held at peak 466 MB on a 64 GB box (ceiling 700 MB), but qualitative render lag at 40-min stress was real. User confirmed the app's target use case is short clips — the multi-hour-stress scenario is acknowledged-imperfect and explicitly not on the optimization roadmap. Mitigations would be (i) two-task split: hot visible-only renderer + cold prefetch, (ii) coarser default density at high duration, (iii) cap visible cells via virtualized rows. None of these are planned

- **(Post-Phase 5 polish plan)** **WebP dropped from SFV's planned format list.** ImageIO has no WebP write support on macOS through 26.5 — runtime-probed via `swift /tmp/probe.swift`: `CGImageDestinationCopyTypeIdentifiers()` returns 22 writable types and `org.webmproject.webp` is not among them (HEIC/JPEG/PNG/TIFF are). `UTType.webP` exists since macOS 11 but is decode-only. Apple DevForum #688001 documents this back to 2021; no encoder mentioned in macOS 14/15/26 release notes. CropBatch's `Models/ExportSettings.swift` lists `.webp` and `ImageCropService.encode/save()` calls `CGImageDestinationCreateWithURL(..., UTType.webP.identifier, …)` directly with no vendored encoder (Package.resolved has only Sparkle) and no fallback — the destination create returns nil, the error is logged at `CropBatchApp.swift:368` and silently swallowed. CropBatch's README "WebP" claim is hollow; no `.webp` file is ever written. SFV's plan ships PNG/JPG/HEIC; vendoring [SDWebImageWebPCoder](https://github.com/SDWebImage/SDWebImageWebPCoder) (~200 KB libwebp) documented as a deferred option if WebP becomes non-negotiable later — adds notarization/hardened-runtime burden, defensible only if asked. See `POLISH_PLAN_post_phase5.md` Item 2.

- **(Post-Phase 5 polish plan)** **JPEG file extension hardcoded to `.jpg`, overriding `UTType.preferredFilenameExtension`.** `UTType.jpeg.preferredFilenameExtension` returns `"jpeg"`, but convention (and macOS Finder norms) is `.jpg`. `ExportFormat.fileExtension` in the plan maps `.jpeg → "jpg"` explicitly. Other formats use the preferred extension as-is (`.png → "png"`, `.heic → "heic"`).

- **(Phase 5 — sorted manual list)** **Manual list is sorted by clip time on insert, not in capture order.** Both `captureCurrentFrame()` and `captureFrame(at:)` route through a private `insertManualTime(_:)` helper that finds the sorted position via `firstIndex { CMTimeGetSeconds($0) > CMTimeGetSeconds(time) }` and `insert(_:at:)`. Reason: the right-pane "Manual Timestamps" list is a destination preview — row N corresponds to filename `_N`. The aspirational "filename index preserves *requested* order" decision was overridden in practice by the existing ascending sort in `TimeListGenerator.times(for: .timestamps(_:))`, so aligning the UI to that is simpler than removing the upstream sort (which `images(for:)` benefits from for sequential GOP decoding anyway). Side effect of the helper: `captureCurrentFrame()` now also dedupes (it didn't before)

- **(Post-Phase 5 polish — xcodegen)** **xcodegen regen IS required for new files in this repo.** The sources glob in `project.yml` is resolved at `xcodegen generate` time, not at `xcodebuild` time. Adding a new `.swift` file under `01_Project/ScreenshotFromVideos/` does NOT auto-include it in the build — `xcodebuild` will fail with "cannot find type X." Run `cd 01_Project && xcodegen generate` before the next build whenever a file is added (or moved). Verified empirically during Item 2 implementation (the new `Models/ExportFormat.swift` triggered exactly this failure on first build). `POLISH_PLAN_post_phase5.md`'s "auto-discovers — no `xcodegen generate` needed" claim was wrong; this entry is the authoritative correction

- **(Post-Phase 5 polish — Item 2 format section placement)** **Format section sits between Output and Options in RightPaneView, not nested inside the Options DisclosureGroup.** Mode → Params → Output → Format → Options reads as "what / when / where / how" in left-to-right precedence. Format materially changes file size and on-disk extension, so it belongs at the top level alongside the destination folder — not nested under "Options" (which is about cosmetic processing like overlay and numbering)

- **(Post-Phase 5 polish — quality slider visibility)** **Quality slider is hidden, not disabled, when PNG is selected.** Implemented as a SwiftUI `if vm.exportFormat.supportsCompression { ... }` branch inside `formatSection`. Rationale: PNG is always lossless, the slider would have no meaning if shown grey'd-out. The layout shift is small (slider row is single-line). Matches CropBatch's UX pattern
